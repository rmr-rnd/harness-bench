# 06 — Architecture Decision Records

ADR фиксируют **почему** система устроена именно так. Используются вместе с C4-диаграммами.

---

## ADR-001: Агент запускается в Docker, а не in-process

**Статус:** Принято

**Контекст:**  
Агентские фреймворки (Hermes, OpenClaw, OpenCode) — самостоятельные процессы со своими зависимостями. Запуск нескольких агентов в одном Python-процессе создаёт конфликты.

**Решение:**  
Каждый агент — `docker run --rm`. Оркестратор управляет через `docker exec` (curl к внутреннему API) и `docker stop`. Контейнер живёт на время одной задачи (или нескольких ходов одной задачи).

**Последствия:**
- `+` Полная изоляция; N задач = N независимых контейнеров
- `+` Версия агента меняется через один параметр `hermes_image`
- `−` Overhead 10–30 сек на pull/start/stop на каждую задачу

## ADR-002: MCP HTTP Bridge между агентом и sandbox

**Статус:** Принято

**Контекст:**  
Агент в Docker должен вызывать инструменты внутри sandbox-контейнера. HTTP через `host.docker.internal` — стандартный способ, который агентские фреймворки поддерживают из коробки. Альтернатива stdio потребовала бы дополнительного процесса-прокси для каждой задачи.

**Решение:**  
На хосте поднимается FastAPI-сервер (`MCPHttpServer`) на случайном порту. Агент обращается к нему через `http://host.docker.internal:{port}/mcp` по протоколу MCP 2025-11-25 (JSON-RPC 2.0). Сервер транслирует вызовы в `docker exec`.

**Последствия:**
- `+` Агент работает по стандартному MCP — не нужно дорабатывать агент
- `+` Кастомные `SandboxTool`-скрипты добавляются без изменения харнеса
- `−` Один порт на задачу; дополнительный сетевой хоп

## ADR-003: Resume через статус задачи в БД

**Статус:** Принято (обновлено — FS-хранилище удалено)

**Контекст:**  
Длинные прогоны прерываются. Перезапуск с нуля дорог по токенам.

**Решение:**  
Перед запуском задачи вызывается `db.fetch_task_output(run_id, task_id)`. Если запись существует и `status == "done"` — задача пропускается (`event="skip"`). Если БД не настроена — resume не работает, задача выполняется заново.

**Последствия:**
- `+` Resume работает корректно даже при параллельных прогонах
- `+` Не нужно следить за файлами на диске
- `−` Без настроенного PostgreSQL resume недоступен

## ADR-004: Scorer вызывается пока sandbox жив

**Статус:** Принято

**Контекст:**  
Для sandbox-задач оценка требует читать файлы и запускать команды внутри контейнера. После `sandbox.stop()` данные уничтожаются.

**Решение:**  
`scorer(sample, trace, judge, sandbox)` вызывается **до** `_teardown_ctx()`. Sandbox передаётся прямо в Scorer как аргумент.

**Последствия:**
- `+` Scorer имеет полный доступ к FS контейнера
- `−` Teardown нельзя переставить до вызова Scorer

## ADR-005: Benchmark и Harness — независимые плагины

**Статус:** Принято

**Контекст:**  
Нужна возможность комбинировать любой агент с любым бенчмарком.

**Решение:**  
Benchmark и Harness — независимые иерархии. Orchestrator связывает их через `ExecutionContext` и `Runner`. Единственный контракт совместимости: `harness.supports_sandbox` должен быть `True` для задач с `sample.sandbox`.

**Последствия:**
- `+` Добавление нового Harness не требует изменений в Benchmark
- `−` Несовместимость (harness без sandbox + sandbox-бенчмарк) проявляется в runtime

## ADR-006: Параллелизм через asyncio + Semaphore

**Статус:** Принято

**Контекст:**  
Задачи должны выполняться параллельно, но ограниченно.

**Решение:**  
`asyncio.Semaphore(workers)` + `asyncio.gather(*tasks, return_exceptions=True)`.

**Последствия:**
- `+` Один параметр `workers` контролирует параллелизм
- `+` Graceful stop: `Orchestrator.stop()` отменяет всё и останавливает sandbox
- `−` CPU-bound операции блокируют event loop → `asyncio.to_thread()`

## ADR-007: BFCL использует in-process mock-бэкенды

**Статус:** Принято

**Контекст:**  
BFCL multi-turn требует реальной обратной связи от "сервисов" (music, weather, …). Поднимать настоящие сервисы дорого и нестабильно.

**Решение:**  
`BFCLMCPServer` запускается in-process на хосте с mock-бэкендами как Python-классами. Агент взаимодействует через стандартный MCP, не зная что это моки.

**Последствия:**
- `+` Детерминированное тестирование, нет сетевых зависимостей
- `+` Memory-снапшот между сессиями через FS
- `−` Mock-бэкенды нужно поддерживать в синхронизации с описаниями функций

## ADR-008: PostgreSQL как единственное хранилище результатов

**Статус:** Принято (обновлено — FS-хранилище удалено)

**Контекст:**  
Изначально использовалось двойное хранилище (FS + PostgreSQL). FS-файлы создавали второй источник правды и усложняли resume-логику.

**Решение:**  
Единственное хранилище — PostgreSQL (asyncpg). Если БД не настроена или недоступна — все методы `Database` работают как no-op, результаты нигде не сохраняются. Схема авто-мигрируется при старте через SQL-файлы из `db/migrations/` (advisory lock для защиты от гонок при параллельном запуске).

Таблицы: `runs`, `benchmark_runs`, `task_outputs` (статус running/done/error/timeout), `eval_results`, `schema_migrations`.

**Последствия:**
- `+` Один источник правды
- `+` SQL-анализ и сравнение прогонов из коробки
- `+` Транзакционный resume через `status == "done"`
- `−` Без PostgreSQL результаты не сохраняются и resume недоступен

## ADR-009: Runner и Scorer как отдельные абстракции

**Статус:** Принято

**Контекст:**  
Изначально харнес управлял и диалогом, и оценкой. Это делало невозможным переиспользование логики оценки между харнесами и многоходовую оркестрацию без дублирования.

**Решение:**  
Разделение на три роли:
- **Harness** — только транспорт: отправить сообщения, получить ответ (`send_turn → TurnResponse`)
- **Runner** — управляет ходами: вызывает `send_turn()` нужное количество раз, агрегирует в `AgentTrace`
- **Scorer** — оценивает результат: получает `Sample + AgentTrace + sandbox → Score`

Benchmark выбирает Runner и Scorer через `make_runner()` / `make_scorer()`.

**Последствия:**
- `+` Харнес не знает о структуре диалога бенчмарка
- `+` BFCL Runner работает с любым харнесом без изменений
- `+` Scorer переиспользуется между бенчмарками
- `−` PAC1 не вписывается — оставлен на legacy `run_task()` протоколе

## ADR-010: Lazy-старт контейнера и ctx.cleanup_fns

**Статус:** Принято

**Контекст:**  
При современном `send_turn()` протоколе Runner вызывает харнес несколько раз. Контейнер нужно запустить один раз при первом вызове и остановить после последнего.

**Решение:**  
Харнес стартует контейнер лениво при первом `send_turn()` и сохраняет `{container_id, previous_response_id}` в `ctx.extras["harness_session"]`. Cleanup-колбэк (`docker stop`) регистрируется в `ctx.cleanup_fns`. Orchestrator вызывает все `cleanup_fns` в `_teardown_ctx()`.

**Альтернативы:**
- Context manager на уровне Orchestrator — усложнил бы интерфейс harness
- Один контейнер на задачу с явным start/stop — нарушило бы абстракцию (Runner не знает о контейнерах)

**Последствия:**
- `+` Харнес полностью инкапсулирует управление контейнером
- `+` Runner и Scorer не знают о Docker
- `−` Cleanup должен быть правильно зарегистрирован — иначе контейнер не остановится (утечка)
